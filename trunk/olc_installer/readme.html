"Smarty"-Installation

Im Hinblick auf ein einfacheres Update der "Smarty"-Module wurde das "Smarty"-Installations-Verzeichnis geändert von "smarty_2.6.2" zu "smarty" (dann kann man nämlich den Inhalt der neuen "Smarty"-Distribution einfach in das Verzeichnis kopieren.

Bei einer Neu-Installation braucht man nichts zu beachten, das das automatisch angelegt wird.

Bei einer Update-Installation muss man den Ordner "smarty" der Distribution in das Shop-Verzeichnis kopieren. Der Ordner "smarty_2.6.2"  kann dann gelöscht werden-

WICHTIG:  Da für die zentrale Unterstützung der AJAX-Technologie die "Smarty"-"fetch"- und -"assign"-Routinen modifiziert wurde, müssen diese Änderung bei neuen "Smarty"-Versionen immer analog modifiziert werden!
 

Präfix für die Datenbank-Tabellen

Mit dieser Version von OL-Commerce ist es möglich, die Datenbank-Tabellen bei der Installation mit einem "Präfix" versehen zu lassen (z.B. "olc_"). Die Tabellennamen heißen dann z.B. "olc_products"

Der Vorteil davon ist, dass man in einer MySQL-Datenbank mehrere OL-Commerce -Shops betreiben kann, oder auch nur die OL-Commerce -Tabellen in Datenbanken, in der sich auch die Daten anderer Anwendungen befinden, besser identifizieren kann.

Bei einer Neu-Installation kann man den gewünschten Präfix gleich angeben, und es wird alles automatisch konfiguriert.

Technisch ist das so gelöst, dass im Modul "includes/database_tables.php" die Definition der Tabellennamen-Konstanten um den definierten Präfix erweitert werden.

Dazu ist es dann aber zwingend erforderlich, dass Datenbank-Tabellen in den Programmen nur noch über diese definierten Konstanten angesprochen werden! (Leider war das bei den Programmen früherer Versionen nicht immer der Fall, so dass diesbezüglich einige Änderungen notwendig waren.)

Wenn man die neue Software mit einer bestehenden Datenbank verwenden will, muss man folgendes tun:

1.   Der Inhalt der Datei "prefix_rename_tables.sql" ist mit PHPmyAdmin auf die bestehende Datenbank anzuwenden.

Zuvor ist der dort eingetragene Präfix "prf_" unbedingt zu dem gewünschten zu ändern (er kann natürlich auch beibehalten werden).
 
2. In den Dateien

includes\configure.php
admin\includes\configure.php

ist der folgende Befehl einzutragen:

define('TABLE_PREFIX','prf_');

(wobei für "prf_" der in 1. gewählte Präfix zu verwenden ist).

In "rename_with_prefix.sql" sind alle Tabellen enthalten, die standardmäßig mit dieser OL-Commerce -Version installiert werden.

Wenn Sie weitere Tabellen in Ihrer Datenbank haben, müssen diese natürlich analog umbenannt werden!

Zusätzlich müssen diese in:

includes/database_tables.php
analog eingetragen werden.

Wenn Sie weitere Programme außer den mit dieser Version installierten verwenden, müssen Sie überprüfen, ob diese die Datenbanktabellen nur über die in "includes/database_tables.php" definierten Namen ansprechen.

Es sind dazu alle Vorkommen der SQL-Befehle

INSERT INTO
SELECT FROM
DELETE FROM
UPDATE
JOIN
in diesen Modulen daraufhin zu überprüfen.

Wenn Sie die Datenbank einer Version 1.2a auf den Bedarf der Version 2.0a/AJAX aktualisieren müssen, dann können Sie "prefix_olcommerce_update_1.2a_2.0a.sql" mit PHPmyAdmin auf die bestehende Datenbank anwenden.

Zuvor ist der dort standardmäßig eingetragene Präfix "prf_" unbedingt zu dem gewünschten zu ändern.
 
  Wenn Sie die OL-Commerce -Tabellen aus der Datenbank löschen wollen, dann können Sie "prefix_delete_tables.sql" mit PHPmyAdmin auf die bestehende Datenbank anwenden.

Zuvor ist der dort standardmäßig eingetragene Präfix "prf_" unbedingt zu dem gewünschten zu ändern.
 

Preisinformation bei jedem Preis (Forderung der Preisangabenverordnung (PAngV)

Im Shop wird durchgängig bei jedem Preis ein Preishinweis angezeigt, der anzeigt, ob/ob nicht die MwSt. enthalten ist, und ob/ob nicht die Versandkosten enthalten sind. Als Sonderfall ist auch die "Kleinunternehmer-Regelung" gemäß §19 UStG möglich.

In dieser Preisinformation wird angezeigt, welche minimalen Versandkosten anfallen, und es wird ein Link zu der Versandkosten-Information des Shops für die Details dazu angeboten (weil i.d.R. ja gewichtsabhängige Versandkosten anfallen werden).

Das Verhalten kann in "lang/german/german.php" konfiguriert werden.

Setzen sie in der Anweisung "define('PRICE_DISCLAIMER_COMMON'...." den minimalen Wert Ihrer Versandkosten ein.

//Setzen Sie die "Versandkosten (ab xxx,-- EUR)"
define('PRICE_DISCLAIMER_COMMON',' zzgl. <a href=javascript:ShowInfo("shop_content.php?coID=1&pop_up=true","") title="Informationen zu den Versandkosten erhalten Sie mit einem Klick hier"><u>Versandkosten (ab 3,-- EUR)</u><br>(Für genauere Info anklicken!)</a>.');

Wenn Sie als Kleinunternehmer §19 UStG im Shop keine MwSt. erheben wollen,  die Zeichen "//" vor der beiden nächsten Anweisungen entfernen!!!

//define('BOX_LOGINBOX_NO_TAX','Gemäß §19 UStG wird keine MwSt. erhoben.');
//define('PRICE_DISCLAIMER_NO_TAX',BOX_LOGINBOX_NO_TAX.'<br>Preis'.PRICE_DISCLAIMER_COMMON);

Sendungscode für Versandtracking

Es besteht die Möglichkeit, beim Versand einer Sendung (Admin-Bereich) den sog. "Tracking-Code" des Versenders mit einzugeben. (Statusänderung der Bestellung auf "Versendet").

Die komplette Tracking-URL (einschl. dem Sendungscode) wird in der Statusänderungs-eMail als Link mitgesendet, so dass der Kunde sehr einfach eine Bestellverfolgung vornehmen kann.

In "lang\german\admin\orders.php" muss dazu die Tracking-URL des Versenders definiert werden.

// 'TRACKURL' -- Hier muss die Tracking-URL des Versenders stehen.

// Tracking Beispiel DHL (POST AG)

// An der Stelle, an der der Sendungscode steht, muss das Zeichen # als Platzhalter stehen!

// An der Stelle, an der die PLZ steht, muss das Zeichen @ als Platzhalter stehen!

define('TRACKURL', 'http://nolp.dhl.de/nextt-online-public/set_identcodes.do?lang=de&zip=@&idc=#');

Diese "Tracking-URL" ist abhängig vom Versender, und muss aus der Internet-Seite, über die man beim Versender die Sendungsverfolgung vornehmen kann, ermittelt werden.

Das ist nicht ganz einfach, und erfordert Erfahrung mit dem Aufbau von HTML-Seiten.

Aber über das Support-Forum kann da sicher geholfen werden.

Infos dazu gibt es auch im xtCommerce-Forum.

eMail-Format (HTML, TEXT) auf kundenebene definieren

Das eMail-Format, das der Kunde erhält (HTML, TEXT), wird auf kundenebene festgelegt, nicht mehr auf systemebene!

Mehrere Templates für einen Shop

Wenn Sie mehrere Templates für Ihren Shop haben, und Sie ihren Besuchern anbieten wollen, sich eines dieser Templates auszuwählen, ist folgendes zu beachten:

Im linken Navigationsbereich oben gibt es den neuen Link "Shop-Design ändern" (nur, wenn mehr als ein Template vorhanden ist!). Nach dessen Anwahl werden alle verfügbaren Templates (außer dem aktuellen) zur Auswahl angeboten.

Wenn im Verzeichnis "images" Bilder mit den Namen "template.gif" (z.B. "olc.gif") existieren, werden diese mit angezeigt. Damit kann man eine verkleinerte Darstellung des Designs anzeigen lassen ("Preview"). (Das sind verkleinerte Screenshots der Eingangsseite. Eine Bildbreite von 300 Pixel ist ausreichend.)

Das gewählte Template wird in einem "Cookie" gespeichert, so dass der Besucher beim nächsten Besuch gleich mit seinem Lieblingsdesign arbeiten kann.
(Das im Admin-Bereich eingestellte Standard-Template wird nicht verändert, so dass ohne Templateauswahl durch den Anwender immer dieses Standard-Template angezeigt wird.)

Wenn Sie keine Template-Auswahl anbieten wollen, können Sie die "Smarty"-Variable "{$CHANGE_SKIN}" aus der "index.html" des Templates entfernen.

Wenn Sie eine Template-Auswahl anbieten wollen, dann müssen alle Templates für die Belange von AJAX angepasst werden!  (Siehe dazu "ajax_aenderungen.html".)

Zusätzlich sollte im Verzeichnis "templates/xxxxxxx/buttons/german" die Datei "button_admin.gif" vorhanden sein, in der eine Variante des "admin"-Buttons im jeweiligen Design enthalten ist. (Ist diese Datei nicht vorhanden, wird der Standard-Admin-Button verwendet.)

Elm@r - Der elektronische Markt

Die Distribution von OL-Commerce 2.0/AJAX enthält (im Verzeichnis "elmar_olcommerce") auch die Installtionsdateien von Elm@r.

Wenn Sie Elm@r verwenden wollen, muss dieses nach Installation des Shops zusätzlich installiert werden.

Elm@r ist primär dazu konzipiert, das "shopinfo.xml"-Format zu erstellen, das mittlerweile von einer ganzen Reihe von Preisvergleichsanbietern verwendet werden kann (inkl. einem automatischen periodischen Daten-Update durch diese Preisvergleichsanbieter). Diese sind z.B. RockBottom, Shopwahl, Antag, Idealo, PreisRoboter, mentasys, wobestellen, Preissuche, Auspreiser, Pangora und Deutsches Shopverzeichnis.

Es kann aber auch Exportdaten generieren für Froogle, Hardwareschotte, Kelkoo, Pangora, Shopping.com, WEB.DE, Amazon.

Für die Preisvergleichsanbieter, die das "shopinfo.xml"-Format untersützten, bracht man gar nichts zu tun, da das Update regelmäßig automatisch(!) durchgeführt wird.

Für die anderen muss man bei Bedarf (Änderung in den Produkten) die Exportdaten neu erstellen und übermitteln lassen.

Näheres zu Elm@r findet man hier.

BLZ-Aktualisierung anhand der Bundesbank-Daten

Im Bestellbereich kann OL-Commerce 2.0/AJAX Bankverbindungen während der Eingabe direkt validieren!

Dazu ist es allerdings erforderlich, dass die BLZ-Informationen in der Datenbank enthalten sind.

OL-Commerce 2.0/AJAX enthält dazu eine intelligente Funktion, die es erlaubt, diese Datenbanktabelle automatisch an Hand der (vierteljährlich aktualisierten) entsprechenden Daten der Bundesbank zu pflegen.

Nach dessen Aktivierung wird der Ablauf genau erläutert.

Sonderbehandlung von Produkt-Preisen <=0

Ein Produktpreis von "0" bewirkt, dass das Produkt zwar angezeigt wird, aber nicht bestellbar ist!

Hat ein Produkt einen Preis < 0, dann wird zusätzlich die Meldung angezeigt, dass es derzeit ausverkauft ist.

Wenn dann gleichzeitig im Produkt noch ein Erscheinungsdatum angegeben ist, dann wird dieses zusätzlich als voraussichtliches Verfügbarkeitsdatum angezeigt. Wenn dieses Erscheinungsdatum dann erreicht ist, wird das Produkt automatisch wieder bestellbar.

Wartungsarbeiten verwalten

Im Admin-Bereich gibt es den Link "Wartungsarbeiten verwalten".

Der Zweck dieser Funktion ist es, einen OL-Commerce -Shop in den "Wartungszustand" zu versetzen, so dass in dieser Zeit niemand, außer bestimmten Anwendern, mehr Zugriff auf den Shop hat.

Über diesen Menüpunkt werden die zugehörigen Optionen angewählt.

Diese sind:

1. Wartungsabschaltung: AN/AUS Damit wird der Wartungszustand AN- oder AUS-geschaltet.

Im "AN"-Zustand dürfen dann nur Anwender mit den folgenden Eigenschaften den Shop bearbeiten:

  • ihre IP entspricht dem in 2. definierten Wert
     
  • ihre IP ist "127.0.0.1" (lokal auf dem "Heimat-System" darf weiter gearbeitet werden)
     
  • sie haben beim Shop-Aufruf den Parameter "allowmaintenance=xxxxxxxxxx" angegeben (siehe 3.)
    (z.B. "www.meinshop.de/olcommerce/index.php?allowmaintenance=Mein Passwort")
2. Wartungsabschaltung: Diese IP-Adresse zulassen Diese IP-Adresse darf die Website während der Wartungsabschaltung bearbeiten (Administrator)
3. Wartungsabschaltung: Dieses Passwort zulassen Wird der Shop mit dem Parameter "allowmaintenance=xxxxxxxxxx" ("xxxxxxxxxx" = Passwort) aufgerufen, dann darf der Benutzer die Website auch während der Wartungsabschaltung bearbeiten (Administrator).
(Das ist eine Alternative für den Fall, dass Sie keine feste IP-Adresse haben, und auf dem Online-Server arbeiten wollen.)
4. Wartungsabschaltung: Vor der Wartungsabschaltung eine WARNUNG anzeigen
 		 Wenn dieser Wert "true" ist, und der nächste Parameter einen Wert enthält, wird immer ein entsprechender Hinweis eingeblendet (auf den Kunden- und den Admin-Seiten).
(Wenn die Wartungsabschaltung auf  "AN" gesetzt wird, wird dieser Wert automatisch zurückgesetzt.)
5. Wartungsabschaltung: Datum und Dauer der Wartungsabschaltung
 		 Datum und Dauer der Wartungsabschaltung (beliebiger Text, z.B.: 1. Mai 2006, 12:00 Uhr bis 15:00 Uhr)

Dieser Hinweis wird während der "Wartungsabschaltung" angezeigt (und evtl. auch während der "Vorwarnung", wenn 4. auf "true" gesetzt wird).

Ein typisches Szenario für die Verwendung dieser Funktion sieht dann wie folgt aus:

Der Admin setzt in 5. die Information, wann der Shop nicht erreichbar ist.

Wenn er (ab sofort!) eine Vorwarnung vor der eigentlichen Abschaltung geben will, setzt er zusätzlich 4. auf "true".

Wenn der Shop abgeschaltet werden soll, setzt er 1. auf "true".

Wenn der Shop wieder angeschaltet werden soll, setzt er 1. auf "false".

Während der "Vorwarnzeit" werden die Vorwarn-Meldungen angezeigt.

Während der Abschaltung können nur noch Besucher, die die Voraussetzungen gemäß 2. oder 3. erfüllen auf den Shop zugreifen, alle anderen erhalten nur eine Meldung, dass kein Zugriff möglich ist.

Google Sitemap

Im Admin-Bereich gibt es den Link "Google Sitemap".

Dieser dient zur Erstellung einer "Sitemap" für Google, so dass eine bessere Indizierung erfolgen kann.

Ausgabe von Test-Informationen

Im AJAX-Modus ist es nicht mehr möglich, mit dem PHP "echo"-Befehl Test-Informationen auszugeben, weil das AJAX-Javascript nur noch Daten in einem bestimmten Format ausgeben.

Um dies dennoch zu ermöglichen, wurde vorgesehen, dass Daten, die in der "superglobalen" PHP-Variablen "$_SESSION['debug_output']" enthalten sind, am Anfang des Bildschirm-Mittelbereiches angezeigt werden.

Wenn man also Testwerte ausgeben will, dann muss man das mit AJAX wie folgt tun:

statt   

echo "testvariable='".$testvariable."'<br>";

schreibt man

$_SESSION['debug_output'].="testvariable='".$testvariable."'<br>";
 (Unbedingt ".=" verwenden, da sonst immer nur die letzte Ausgabe angezeigt wird!)

Shop-Funktion bei deaktiviertem Javascript

Es war von Anfang an das Ziel, mit einer Version sowohl den konventionellen als auch den AJAX-Modus betreiben zu können.

Wie wird das erreicht?

Wenn das Programm aufgerufen wird, wird zunächst eine Javascript-Erkennung durchgeführt, indem eine kleine HTML-Seite an den Browser zurückgegeben wird.

Bei aktiviertem Javascript kann diese sich aktiv beim Server zurückmelden, und der AJAX-Modus wird aktiviert.

Bei nicht aktiviertem Javascript wird ein PHP-Fehlerskript aufgerufen (mit Hilfe des HTML "<NOSCRIPT>"-Tags), das dem Besucher diese Tatsache mitteilt, und ihm 2 Möglichkeiten anbietet:

1. Er wird aufgefordert, seinen Browser für die Javascript-Verwendung zu konfigurieren.

2. Er kann das Programm im konventionellen Modus weiter betreiben.

Im Fall 1. kann er sich dann eine genaue Anleitung anzeigen lassen, wie das geht.

Er kann dann nach der Konfiguration im AJAX-Modus oder doch konventionell starten.

Das Ganze funktioniert auch, wenn der Besucher über einen "deep link" auf der Shop-Seite landet (z.B. wenn über Froogle & Co. direkt eine Produktseite aufgerufen wurde).

Die "deep link"-Information bleibt auf den Informations-Seiten erhalten, so dass der Besucher über die angebotenen Links im AJAX- oder konventionellen Modus auf der angewählten Seite landet (was im AJAX-Modus gar nicht so einfach zu realisieren war).

Die o.g. Informations-Seiten ("templates/aktuelles template/module/ajax_.......html") sind so konzipiert, dass das CSS des Shop-Templates und das Shop-Logo ("templates/aktuelles template/images/logo.gif") verwendet werden.
